After running `0cmplx test` against a few servers, patterns emerge. These are the vulnerabilities that come up most often, with examples and fixes.

## Excessive data exposure

**The problem**: your tool returns everything the API gives back, including fields the AI model does not need.

```javascript
// Before: returns everything
async function getPet({ id }) {
  const response = await fetch(`${API_BASE}/pets/${id}`);
  return response.json();
  // Returns: { id, name, species, _internalId, ownerId, createdAt, updatedAt, ... }
}
```

The model sees all those fields. It might include `_internalId` or `ownerId` in its response to the user. Worse, it might use `ownerId` as input to another tool, leaking access across users.

**The fix**: filter the response.

```javascript
// After: returns only what the model needs
async function getPet({ id }) {
  const response = await fetch(`${API_BASE}/pets/${id}`);
  const pet = await response.json();
  return {
    id: pet.id,
    name: pet.name,
    species: pet.species,
    age: pet.age,
  };
}
```

Return the minimum set of fields that lets the model answer the user's question.

## Missing authentication

**The problem**: the tool calls your API without credentials, and the API does not enforce auth on certain endpoints.

This is common with generated servers when the OpenAPI spec does not mark every endpoint's security requirements consistently. The `/pets` list might require auth, but `/pets/{id}` might not.

**The fix**: ensure every handler includes authentication, and test with and without credentials:

```javascript
async function deletePet({ id }) {
  const response = await fetch(`${API_BASE}/pets/${id}`, {
    method: 'DELETE',
    headers: {
      'Authorization': `Bearer ${process.env.API_TOKEN}`,
    },
  });

  if (response.status === 401 || response.status === 403) {
    return { error: 'Not authorised to delete this pet.' };
  }

  return { success: true };
}
```

## Parameter injection

**The problem**: a tool parameter is used in a way that allows injection. This is most common when parameters are interpolated into strings.

```javascript
// Dangerous: id could contain path traversal
async function getFile({ id }) {
  const data = fs.readFileSync(`./data/${id}.json`);
  return JSON.parse(data);
}
// An AI model might call this with id = "../../etc/passwd"
```

**The fix**: validate and sanitise parameters before using them.

```javascript
async function getFile({ id }) {
  if (!/^[a-zA-Z0-9_-]+$/.test(id)) {
    return { error: 'Invalid file ID.' };
  }
  const data = fs.readFileSync(`./data/${id}.json`);
  return JSON.parse(data);
}
```

The input schema provides type-level validation (string, number, enum), but it does not prevent valid strings from containing dangerous values. Validation in the handler is still needed.

## Unbounded list responses

**The problem**: a list tool returns all records with no pagination limit.

```javascript
async function listPets() {
  const response = await fetch(`${API_BASE}/pets`);
  return response.json();
  // Returns 10,000 records
}
```

The AI model asked for "a list of pets" and got the entire database. This wastes tokens, slows down the response, and might expose data that should be paginated.

**The fix**: enforce a maximum and let the model paginate.

```javascript
async function listPets({ limit = 20, offset = 0 }) {
  const safeLimit = Math.min(limit, 100);
  const response = await fetch(
    `${API_BASE}/pets?limit=${safeLimit}&offset=${offset}`
  );
  return response.json();
}
```

## Missing error context

**The problem**: when a tool call fails, the handler returns a raw error or stack trace.

```javascript
// Bad: exposes internals
async function createPet({ name }) {
  try {
    const response = await fetch(`${API_BASE}/pets`, { ... });
    return response.json();
  } catch (err) {
    return { error: err.stack };
    // "Error: connect ECONNREFUSED 10.0.0.5:3306\n    at ..."
  }
}
```

The model now knows your database IP, port, and driver. This is information leakage.

**The fix**: return a useful error without internals.

```javascript
catch (err) {
  return { error: 'Could not create the pet. The service may be temporarily unavailable.' };
}
```

Tell the model what happened in terms it can relay to the user. Log the details server-side.

## The pattern

Most vulnerabilities follow the same shape: something crosses a boundary without being checked. Data comes in without validation, data goes out without filtering, errors leak without redaction. The fix is always the same: check at the boundary.
