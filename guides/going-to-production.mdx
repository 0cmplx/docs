---
title: Going to production
description: Checklist for deploying your MCP server with confidence.
---

You have tested your MCP server in the sandbox. Before connecting real data, walk through this checklist.

## Security

<Steps>
  <Step title="Run security traps with zero failures">
    All traps should pass or be acknowledged warnings. No failures.

    ```bash
    0cmplx test --target http://localhost:3100
    ```
  </Step>
  <Step title="Review all warnings">
    Each warning is a judgement call. Document why you are accepting it or fix it.
  </Step>
  <Step title="Check authentication on every tool">
    Call each tool without credentials and verify it is rejected. Test the actual behaviour, not the configuration.
  </Step>
  <Step title="Review response filtering">
    For each tool, check that the response includes only the fields the model needs. Remove internal IDs, timestamps, and metadata unless they serve the user.
  </Step>
</Steps>

## Configuration

<Steps>
  <Step title="Credentials from environment">
    Database credentials and API tokens should come from environment variables, never from configuration files or code.
  </Step>
  <Step title="Set connection limits">
    Configure connection pooling and query timeouts on your database connection. Unbounded connections from a busy MCP server can overwhelm a database.
  </Step>
</Steps>

## Reliability

<Steps>
  <Step title="Error handling on every tool">
    Every tool should catch errors and return a message the model can understand. Never return stack traces or raw error objects.
  </Step>
  <Step title="Response size limits">
    List operations should have maximum page sizes. Large responses waste tokens and slow everything down.
  </Step>
  <Step title="Log tool calls">
    Record what tools were called, with what parameters, and what they returned. This is invaluable for debugging and auditing.
  </Step>
</Steps>

## Deployment

<Steps>
  <Step title="Test with real data in staging">
    Connect to a staging database. Ask questions that exercise your tools. Watch for unexpected behaviour or missing relationships.
  </Step>
  <Step title="Monitor after launch">
    Watch your logs for unusual patterns: repeated failed calls, unexpected parameter values, or tools being called in sequences that do not make sense.
  </Step>
</Steps>

## The short version

1. Zero test failures
2. Warnings reviewed
3. Auth on every tool
4. Responses filtered
5. Credentials from environment
6. Error handling everywhere
7. Tested in staging

If all seven are done, deploy with confidence.
