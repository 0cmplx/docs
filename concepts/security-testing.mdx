An MCP server is an API that AI agents call. That makes it an attack surface. The caller is not a human clicking buttons; it is a language model choosing parameters based on a conversation. The model might be helpful. It might also be confused, hallucinating, or manipulated by a prompt injection attack.

0cmplx's security traps simulate these scenarios. They connect to your MCP server as a client would, call tools with carefully crafted inputs, and check what happens.

## What the traps test

Each trap targets a specific category of vulnerability. They are inspired by OWASP's work on web application security, adapted for the MCP context.

<CardGroup cols={2}>
  <Card title="Injection" icon="syringe">
    Can a tool parameter carry a payload that the server executes or passes through unsanitised? This includes SQL fragments, shell commands, and prompt text designed to alter downstream behaviour.
  </Card>
  <Card title="Excessive data exposure" icon="eye">
    Does a tool response include more data than the caller needs? Internal IDs, email addresses, tokens, or debug information that should not reach the AI model.
  </Card>
  <Card title="Broken access control" icon="lock-open">
    Can a tool be called without proper authentication? Can one user's request access another user's data by changing an ID parameter?
  </Card>
  <Card title="Resource exhaustion" icon="battery-empty">
    Can a tool be called in a way that consumes excessive resources? Unbounded list queries, recursive operations, or missing rate limits.
  </Card>
</CardGroup>

## How a trap works

A trap is a function that:

1. Connects to the MCP server
2. Discovers available tools
3. Calls one or more tools with adversarial inputs
4. Inspects the response
5. Reports pass, warning, or failure

The trap connects to your MCP server (or sandbox), sends adversarial inputs through the tool, and inspects the response. It does not modify your server. If a tool accepts a SQL fragment in a parameter and the response changes in a way that suggests it was executed, the trap flags it.

## Pass, warn, fail

- **Pass**: the server handled the adversarial input safely.
- **Warn**: the behaviour is not clearly dangerous, but worth reviewing. For example, a response includes fields that look like internal identifiers.
- **Fail**: the server exhibited behaviour that is likely exploitable. Immediate attention recommended.

Warnings are not noise. They are the traps saying "I am not sure, but this looks interesting." A warning on data exposure might mean your API returns an `_id` field that is harmless, or it might mean it returns a session token. 0cmplx cannot tell the difference. You can.

## Testing servers you did not build

The traps work against any MCP server, not just ones generated by 0cmplx. If you have a hand-written server, or one generated by another tool, you can point `0cmplx test` at it and get the same report.

This makes 0cmplx useful as a standalone security tool, independent of the generation feature.

## Limitations

Security traps find common problems. They do not find all problems. They cannot reason about your business logic, understand your data model, or know which fields are sensitive in your context.

Think of them as a first pass. They catch the things that are easy to miss and tedious to test manually. They do not replace a security review, but they make the review faster by handling the mechanical parts.
