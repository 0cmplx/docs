The engine does not expose CRUD tools for each table. It exposes a query interface. You ask a question in natural language. The engine composes the SQL.

## Why composition matters

A typical database has dozens of tables with foreign key relationships. To answer "What is the premium for the user with phone +2781?", a CRUD approach needs three tool calls: look up the user, find their policy, fetch the premium. Three round trips. Three LLM reasoning steps.

The engine does it in one. It knows the relationship graph:

```
users.phone → users.id
policies.user_id → users.id
premiums.policy_id → policies.id
```

It composes:

```sql
SELECT p.amount, p.month
FROM premiums p
JOIN policies pol ON p.policy_id = pol.id
JOIN users u ON pol.user_id = u.id
WHERE u.phone = '+2781...'
AND p.month = '2026-05'
```

One tool call. One result.

## The relationship graph

When you connect your data, the engine extracts:

- **Tables** with columns and types
- **Foreign keys** (inline, table-level, and named constraints)
- **Unique constraints** for natural lookups (phone, email, account number)
- **Defaults and nullability** for understanding optional fields

This becomes a graph where nodes are tables and edges are foreign key relationships. The engine traverses this graph to compose joins.

## Confidence scoring

The engine scores how well a question maps to the schema:

**High confidence** (executes immediately):
- All terms resolve to known tables and columns
- Relationships exist to join the required data
- Example: "Get status for user with phone +2781..."

**Low confidence** (returns clarification):
- Ambiguous terms that match multiple columns
- References to data that does not exist in the schema
- Unclear operations ("adjusted", "normalised", "weighted")

```json
{
  "status": "needs_clarification",
  "confidence": 0.34,
  "resolved": ["users.phone", "accounts.balance"],
  "unresolved": ["loyalty tier - no matching column"]
}
```

The caller has the domain knowledge to rephrase. The engine has the schema knowledge to execute. Neither guesses.

## Sandbox execution

Every query runs in a Pyodide sandbox with an ephemeral database:

- **Isolated**: no access to production data or other apps
- **Ephemeral**: cleaned up when the app expires
- **Safe**: generated SQL is executed, not raw user input
- **Observable**: every query is logged with timing and results
