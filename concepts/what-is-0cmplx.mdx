title: What is 0cmplx
description: A research and experimental platform for testing AI agents, MCP servers, and tool integrations.
---

0cmplx is a research and experimental platform. It exists because the space between AI models and the tools they call is still mostly unexplored, and the people building in that space need somewhere to experiment safely.

## What it does today

You connect your AI agent to mock MCP servers that simulate real-world tools: APIs, databases, file systems. Some of those servers are designed to behave well. Others are designed to break your agent, using attack patterns drawn from OWASP research: prompt injection, tool poisoning, excessive agency, data exfiltration.

You watch what happens. You see which attacks land and which your agent resists. You get a security score and a performance profile. Then you iterate.

No production data at risk. No live systems exposed. Just a sandbox where you can push your agent to its limits and learn what it does under pressure.

## What it is becoming

We are building toward a platform where you bring your own API spec or database schema and 0cmplx becomes the execution layer between your LLM and your data.

<CardGroup cols={2}>
  <Card title="API mode" icon="plug">
    Upload an OpenAPI spec. 0cmplx generates typed functions. The LLM writes a script. 0cmplx calls your API, filters responses, and returns only what matters.
  </Card>
  <Card title="SQL mode" icon="database">
    Upload a schema. 0cmplx spins up an ephemeral database. The LLM writes queries. You see exactly what it generates and what comes back, against sample data instead of production.
  </Card>
</CardGroup>

The LLM plans. 0cmplx executes. Any model, any provider, no lock-in.

## Who it is for

**Developers building AI agents.** You have an agent that calls tools. You want to know if it handles adversarial inputs. You want to test it against MCP servers that fight back.

**Developers building MCP servers.** You have a server. You want to know if it leaks data, accepts injected parameters, or exposes too much surface area. You want OWASP traps to probe it before your users do.

**Researchers and experimenters.** You are curious about how AI models interact with tools. You want to try different models against the same test suite and compare behaviour. You want a controlled environment for exploring questions that do not have answers yet.

## What we believe

The MCP protocol opened a door. AI models can now discover and call tools in a standardised way. But the ecosystem is young. Most MCP servers are built by hand, untested, and deployed without security review.

We think the interesting questions are not about the protocol itself. They are about what happens when you put a language model on one end and a real API on the other:

- How does the model choose between 5 tools versus 50?
- What happens when a tool description contains hidden instructions?
- Can the model be tricked into calling a destructive endpoint?
- Does the model leak sensitive data from tool responses back to the user?
- How do you test all of this before it reaches production?

0cmplx is where you explore these questions. Not in theory, but by running experiments.

## What is coming

<Steps>
  <Step title="Relationship-aware context">
    When you upload a spec or schema, 0cmplx builds a graph of how your endpoints and tables relate. Instead of showing the LLM everything, it surfaces only the relevant cluster per question. Fewer tokens, better tool selection, more accurate results.
  </Step>
  <Step title="Execution streaming">
    Watch the LLM write a script token by token. See each API call and SQL query execute in real-time. Full visibility into the execution pipeline, not just the final answer.
  </Step>
  <Step title="Organisation sandboxes">
    Dedicated subdomains per organisation. Bring your team, share test suites, compare agent behaviour across models. Eventually, connect your own infrastructure.
  </Step>
  <Step title="Community test suites">
    Publish and share OWASP trap configurations. Build on what others have tested. Contribute attack patterns back to the community.
  </Step>
</Steps>

## The name

0cmplx. Zero complexity. The goal is to make testing AI tool integrations as simple as uploading a spec and asking a question. The complexity is in the sandbox, not in your workflow.
