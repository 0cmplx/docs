The LLM decides what to do. 0cmplx does it.

0cmplx Execute is a sandbox that runs scripts written by any LLM. You upload an OpenAPI spec or a database schema. 0cmplx generates typed functions from it. The LLM writes a Python script using those functions. 0cmplx executes the script, calls your API or queries the database, and returns processed results.

No round-trips to the model between tool calls. No provider lock-in. No credentials exposed to the LLM.

## API mode

You have an API. You have a spec for it. 0cmplx reads the spec and generates typed functions the LLM can use.

### Upload your spec

Drag in an OpenAPI 3.x spec (JSON or YAML) or a Postman collection. 0cmplx parses every endpoint and builds a **relationship graph**: which endpoints are related by path hierarchy, shared parameters, and response types.

When the user asks a question, 0cmplx does not dump all 47 endpoints into the LLM's context. It identifies the relevant cluster and surfaces only those functions:

```
Your question involves pets and their owners.

  pets
    listPets(limit?: number, species?: string) → Array<Pet>
    getPet(id: number) → Pet
    createPet(name: string, species: string) → Pet

  owners (related via Pet.owner_id)
    getOwner(id: number) → Owner
    listOwnerPets(owner_id: number) → Array<Pet>

  Types:
    Pet: id, name, species, age, owner_id
    Owner: id, name, email
```

5 functions instead of 47. The LLM picks the right tools because it only sees relevant ones. It never sees your base URL, auth headers, or raw HTTP responses.

### The LLM writes a script

Given a user's question and the available functions, the LLM writes a Python script:

```python
# "How many dogs do we have, and what's the oldest?"
pets = await listPets(limit=100, species="dog")
oldest = max(pets, key=lambda p: p['age'])
print(f"Total dogs: {len(pets)}")
print(f"Oldest: {oldest['name']}, {oldest['age']} years old")
```

One script. No round-trips. The LLM writes it once and hands it to 0cmplx.

### 0cmplx executes

The script runs in 0cmplx's sandbox. Each function call becomes a real HTTP request to your API:

```
listPets(limit=100, species="dog")
  → GET https://api.example.com/pets?limit=100&species=dog
  → Authorization: Bearer ••••••
  → Response filtered to: [{id, name, species, age}, ...]
```

0cmplx handles authentication, parameter validation, timeouts, and response filtering. The LLM receives only the script's printed output:

```
Total dogs: 47
Oldest: Biscuit, 14 years old
```

## SQL mode

Sometimes the question is about data, not endpoints. SQL mode lets the LLM query a database directly.

### Upload your schema

Provide your CREATE TABLE statements:

```sql
CREATE TABLE owners (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  email VARCHAR(255),
  created_at DATETIME
);

CREATE TABLE pets (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  species VARCHAR(50),
  age INT,
  owner_id INT REFERENCES owners(id)
);
```

0cmplx spins up an ephemeral database and builds a **relationship graph** from foreign keys. When the user asks a question, 0cmplx traces the FK chains and surfaces only the relevant tables.

### The LLM sees related tables, not all tables

If the user asks about orders and revenue, 0cmplx follows the FK chain and surfaces:

```
Your question involves orders and products.

  customers (id, name, email, region)
    → orders (id, customer_id FK, total, status, placed_at)
      → order_items (id, order_id FK, product_id FK, quantity, price)
        → products (id, name, category, price)

4 tables. Not shown: audit_log, sessions, email_templates (no FK path to this cluster).
```

The LLM writes better SQL because it sees the joins that matter and nothing else.

### The LLM writes SQL

```sql
SELECT o.name AS owner, COUNT(p.id) AS pet_count, MAX(p.age) AS oldest_pet
FROM owners o
JOIN pets p ON p.owner_id = o.id
GROUP BY o.id
ORDER BY pet_count DESC
LIMIT 5;
```

### 0cmplx executes

The query runs against the ephemeral sandbox DB. Results come back as structured data. The LLM never has a connection string, never sees credentials, never touches your production database.

## Relationship-aware context

This is the core design principle. 0cmplx is not a general-purpose sandbox that dumps everything into the LLM's context window.

When you upload a spec or schema, 0cmplx builds a relationship graph:

- **API mode**: path hierarchy (`/pets/{id}/vaccinations` relates to `/pets`), shared parameters (`customer_id` links orders to customers), response types (`listPets` and `getPet` both return `Pet`)
- **SQL mode**: foreign keys (`orders.customer_id → customers.id`), transitive joins (customers → orders → order_items → products)

When a question arrives, 0cmplx identifies the relevant entities, traces the graph, and surfaces only the related cluster. Tables and endpoints with no path to the cluster are excluded.

This means:
- **Fewer tokens**: the LLM sees 5 functions instead of 47, 4 tables instead of 30
- **Better accuracy**: the LLM picks the right tools because irrelevant ones are not there to confuse it
- **Better SQL**: the LLM sees the FK chain and writes correct joins without guessing

A general-purpose sandbox says "here are all your tools." 0cmplx says "here are the tools that matter for this question."

This is ideal for:
- **Validating queries** before running them against production
- **Exploring schemas** to understand relationships and data shapes
- **Testing security traps** against LLM-generated SQL (injection, exfiltration, writes)

### API mode: end to end

A weather API with an OpenAPI spec. The user asks a question that requires multiple endpoint calls.

**1. Upload the spec**

```bash
0cmplx execute init --spec ./weather-api.yaml --auth bearer:$WEATHER_API_KEY
```

0cmplx parses the spec and reports what it found:

```

  Functions generated:
    getCurrentWeather(city: string) → WeatherData
    getForecast(city: string, days?: number) → Array<ForecastDay>
    getHistorical(city: string, date: string) → WeatherData
    listAlerts(region: string) → Array<Alert>

  Types:
    WeatherData: temp (number), humidity (number), conditions (string), wind_speed (number)
    ForecastDay: date (string), high (number), low (number), conditions (string), rain_chance (number)
    Alert: id (string), severity (string), message (string), expires (string)

  Auth: Bearer token configured
  Ready to execute.
```

**2. Ask a question**

```bash
0cmplx execute run --prompt "Compare today's weather in London, Paris, and Tokyo. Which city is warmest? Also check if there are any weather alerts for Europe."
```

**3. The LLM writes this script** (generated automatically, shown for transparency):

```python
# Fetch weather for all three cities in parallel
import json

cities = ["London", "Paris", "Tokyo"]
weather = {}
for city in cities:
    data = await getCurrentWeather(city)
    weather[city] = data

# Find the warmest
warmest = max(weather.items(), key=lambda x: x[1]['temp'])
print(f"Current temperatures:")
for city, data in weather.items():
    print(f"  {city}: {data['temp']}C, {data['conditions']}")
print(f"\nWarmest: {warmest[0]} at {warmest[1]['temp']}C")

# Check European alerts
for region in ["UK", "Western Europe"]:
    alerts = await listAlerts(region)
    if alerts:
        print(f"\nAlerts for {region}:")
        for alert in alerts:
            print(f"  [{alert['severity']}] {alert['message']}")
    else:
        print(f"\nNo alerts for {region}")
```

An e-commerce database schema. The user wants to explore sales data.

    order_items (id INT PK, order_id INT FK→orders.id, product_id INT FK→products.id, quantity INT, price DECIMAL)
    products (id INT PK, name VARCHAR, category VARCHAR, price DECIMAL, stock INT)

  Ephemeral DB created (SQLite)
  Seeded with sample-data.sql (1,200 customers, 8,400 orders, 34,000 items, 150 products)
  Ready to execute.

**2. Ask a question**

```bash
0cmplx execute run --prompt "Which product category has the highest revenue this quarter? Break it down by region. Also flag any customers who ordered more than 10 times."
```

**3. The LLM writes this script**:

```python
# Revenue by category and region for current quarter
revenue = await query("""
    SELECT p.category, c.region,
           SUM(oi.quantity * oi.price) AS revenue,
           COUNT(DISTINCT o.id) AS order_count
    FROM order_items oi
    JOIN orders o ON o.id = oi.order_id
    JOIN products p ON p.id = oi.product_id
    JOIN customers c ON c.id = o.customer_id
    WHERE o.placed_at >= date('now', 'start of month', '-2 months')
      AND o.status != 'cancelled'
    GROUP BY p.category, c.region
    ORDER BY revenue DESC
""")

# Print category breakdown
print("Revenue by category and region (this quarter):")
print(f"{'Category':<20} {'Region':<15} {'Revenue':>12} {'Orders':>8}")
print("-" * 58)
for row in revenue:
    print(f"{row['category']:<20} {row['region']:<15} ${row['revenue']:>11,.2f} {row['order_count']:>7}")

# Top category overall
from collections import defaultdict
totals = defaultdict(float)
for row in revenue:
    totals[row['category']] += row['revenue']
top = max(totals.items(), key=lambda x: x[1])
print(f"\nTop category: {top[0]} (${top[1]:,.2f} total)")

# Frequent customers
frequent = await query("""
    SELECT c.name, c.email, COUNT(o.id) AS order_count
    FROM customers c
    JOIN orders o ON o.customer_id = c.id
    WHERE o.placed_at >= date('now', 'start of month', '-2 months')
    GROUP BY c.id
    HAVING COUNT(o.id) > 10
    ORDER BY order_count DESC
""")

if frequent:
    print(f"\nCustomers with >10 orders this quarter:")
    for row in frequent:
        print(f"  {row['name']} ({row['email']}): {row['order_count']} orders")
else:
    print("\nNo customers with >10 orders this quarter.")
```

## The sandbox

The sandbox is the core of 0cmplx Execute. It:

1. **Receives a script** from the LLM (Python subset with generated functions)
2. **Executes it** against your API or the ephemeral DB
3. **Enforces security**: credentials injected at runtime (never in the script), parameter validation, rate limits, response filtering, read-only DB by default
4. **Logs everything**: what was called, with what parameters, what came back, how long it took
5. **Returns only output**: the LLM sees what the script printed, not the raw API responses or query results

### Python subset

The sandbox runs a restricted Python environment. The LLM can use:
- Generated functions (from your spec or schema)
- Standard data processing: list comprehensions, sorting, aggregation, string formatting
- Safe standard library: `json`, `math`, `datetime`, `collections`

The sandbox blocks:
- File I/O
- Network access (beyond the configured API)
- Dangerous imports (`os`, `subprocess`, `socket`)
- Unbounded loops (timeout enforcement)

Every LLM can write Python. The sandbox does not care which model wrote the script.

## Why 0cmplx owns the sandbox

Some AI providers offer their own code execution environments. The model writes code, the provider runs it. This works, but it means:

- **You are locked to that provider.** Switch models and you lose the execution capability.
- **The provider sees your data.** API responses and query results pass through their infrastructure.
- **You depend on their audit trail.** If the provider does not log tool calls, you have no visibility.
- **Their sandbox, their rules.** Timeouts, resource limits, and available libraries are not yours to configure.

0cmplx's sandbox is yours. It runs on your terms, logs on your terms, and works with whatever LLM you choose. The model writes a script. The script is just text. Any model can produce it.

## Security traps for both modes

0cmplx's OWASP traps adapt to each execution mode:

**API mode traps**:
- Injection through function parameters
- Excessive data in API responses (before 0cmplx filters them)
- Authentication bypass (calling without credentials)
- Rate limit exhaustion from scripted loops

**SQL mode traps**:
- SQL injection in LLM-generated queries
- Schema enumeration (accessing tables outside the uploaded schema)
- Write operations when read-only mode is configured
- Unbounded SELECT queries (missing LIMIT, SELECT *)

The traps test the sandbox boundary. They simulate what a manipulated LLM might produce and check whether 0cmplx catches it.
