---
title: Why your own sandbox
description: Why 0cmplx runs scripts on your infrastructure instead of inside a provider's container.
---

Some AI providers offer code execution as a built-in feature. The model writes code, the provider runs it. This is useful for general computation, but 0cmplx takes a different approach for good reasons.

## Credentials stay on your server

When a provider runs code on your behalf, tool results pass through their infrastructure. They see the data.

With 0cmplx, the sandbox runs on your server. The bridge calls your API directly. Credentials are injected server-side. The LLM only sees the script's printed output, never the raw API responses.

## Any model, any provider

Provider sandboxes are tied to one model family. Switch providers and you lose the execution capability.

0cmplx scripts are just text. Any model can write Python. The sandbox does not care which model produced the script.

## Spec-driven, not manual

Provider sandboxes start empty. You define tools manually as JSON schemas with `allowed_callers` configurations.

0cmplx starts with your API spec or database schema. Functions are generated. Relationship graphs are built. The model sees only what's relevant.

## Security testing built in

Provider sandboxes do not test your tools for vulnerabilities. 0cmplx runs OWASP traps against the sandbox boundary: injection, data exposure, auth bypass, resource exhaustion.

## When a provider sandbox makes sense

General-purpose computation: data science, file processing, chart generation, complex maths. If the task is about crunching data, not about calling your API, a provider sandbox is the right tool.

0cmplx is for the other case: when the model needs to interact with your API and your data, safely, with observability, and without locking you to a provider.
